<objective>
Wire VibeDrift in as GSD's SECOND upstream (D-01/D-02): the same relationship gsd-plugin has with gsd-core. GSD never shells out to `vibedrift` at runtime (D-01) — instead a standalone cron-style notifier watches `@vibedrift/cli` for new releases so the maintainer learns when there are new heuristics worth porting natively (the porting roadmap in CONTEXT §specifics).

Per RESEARCH Open Question 3, a small standalone sibling of `check-gsd-release.sh` is cleaner than overloading the gsd-core watch. This plan ships `bin/check-vibedrift-release.sh` modeled exactly on the existing watch, then (via a human-action checkpoint) the maintainer installs it into cron, syncs the live copy, and seeds `~/.vibedrift-last-known-version` so the first run fires no spurious email (RESEARCH §Runtime State Inventory; memory `260608-vk9` — the cron runs a COPY at `~/claude-code-gsd`, not a symlink).

This plan has NO file overlap with 11-01/11-02/11-04/11-05, so it runs in parallel in Wave 1. It is `autonomous:false` because the cron install + live-copy sync + seed touch the maintainer's machine state (no CLI substitute).
