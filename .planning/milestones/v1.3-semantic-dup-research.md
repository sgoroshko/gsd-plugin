# Semantic Duplication Detection — Research & MVP Spec

**Researched:** 2026-06-26
**Domain:** Type-4 (semantic) code clone detection for a mixed JS/CJS/TS + markdown repo
**Confidence:** MEDIUM-HIGH (tooling/versions verified; threshold guidance is empirical and model-specific)

---

## 1. The problem, located precisely

Clone taxonomy (standard literature, e.g. Roy/Cordy, the 2023 systematic review arXiv:2306.16171) [CITED: arxiv.org/pdf/2306.16171]:

| Type | What | Example | Caught by |
|------|------|---------|-----------|
| **Type-1** | Identical text (whitespace/comments aside) | copy-paste | any tool |
| **Type-2** | Identical structure, renamed identifiers/literals | `a+b` vs `x+y` | token tools (jscpd, CPD, Simian) |
| **Type-3** | Near-miss: statements added/removed/reordered | a clone + 2 extra lines | AST/structural tools, SourcererCC, NiCad |
| **Type-4** | **Same behavior, different syntax & structure** | a `for` loop vs `.reduce()` that compute the same thing | **only semantic/embedding/LLM methods** |

GSD's failure mode — session B re-implements session A's logic in a different shape — is **squarely Type-4**. The two functions may share *zero* tokens of meaningful overlap. This is the single most important framing fact: **token and AST tools cannot see this class of duplicate.** [CITED: arxiv.org/pdf/2109.12079 — SEED, Type-4 is "the hardest to detect"]

## 2. Approaches surveyed (with honest tradeoffs)

### Token / AST structural tools — `jscpd`, PMD CPD, Simian
- **Catch:** Type-1, Type-2 reliably; jscpd uses Rabin-Karp on token streams. [CITED: dev.to/rahulxsingh duplicate-code-checkers-2026]
- **Do NOT catch:** Type-3 partially, **Type-4 not at all.** Token tools are language-agnostic and fast but blind to logical equivalence. [VERIFIED: cross-confirmed jscpd/CPD/Simian all token-based, Type-1/2 only]
- **Verdict for us:** Useful as a *cheap exact-clone pre-pass* (run jscpd to strip obvious copy-paste so embeddings focus on the interesting cases), but **insufficient as the detector.** jscpd 5.0.11 is current. [VERIFIED: npm registry]

### AST-normalization / structural hashing
- Normalize identifiers/literals → hash subtrees → equal hash = clone. Catches Type-2 and some Type-3. Still **structure-bound**: a loop vs a `.reduce()` produce different trees, so Type-4 escapes. Good complexity/cost but wrong target for our problem.

### Embedding-based (RECOMMENDED CORE)
- Embed each function with a code-embedding model → cosine-similarity nearest-neighbors → flag high-similarity pairs that are **not** textually identical. This is the only family that scales to Type-4 without per-pair LLM cost. [CITED: arxiv.org/pdf/2507.16661 VulCoCo uses exactly embed→cosine→threshold]
- **Model/dependency options (the honest comparison):**
  | Option | What | Offline? | Cost | Quality | Notes |
  |--------|------|----------|------|---------|-------|
  | **`jina-embeddings-v2-base-code`** (local, ONNX via `@huggingface/transformers`) | 161M-param code model, 30 languages, 8192-token ctx | **Yes** | free after ~300MB download | Strong for code-vs-code | **MVP pick.** q8 quant runs on CPU. [CITED: huggingface.co/jinaai/jina-embeddings-v2-base-code] |
  | Hosted embeddings API (Voyage `voyage-code-3`, OpenAI `text-embedding-3`) | best-in-class accuracy | No | per-token $ + network | Highest (Voyage tops code benchmarks) | Better recall, but adds a key + network; defer. [CITED: 2025 benchmark coverage] |
  | Claude / LLM as embedder | n/a — Claude has no embeddings endpoint | No | — | — | Not an embedding option; Claude's role is the *judge* (below) |
  - **Lightest thing that works offline-ish:** local `jina-embeddings-v2-base-code` at q8 quant through `@huggingface/transformers` (Transformers.js, successor to `@xenova/transformers`). No server, no API key, pure Node. [VERIFIED: npm `@huggingface/transformers@4.2.0`]

### LLM-judge (pairwise "are these equivalent?")
- High precision on the *hard* pairs, but **O(n²)** over all functions is infeasible (~3k functions → ~4.5M pairs). LLM-as-judge also shows order-sensitivity/inconsistency on SE tasks. [CITED: arxiv.org/pdf/2502.06193]
- **Correct use:** only as a *precision filter on the top-K embedding candidates*, never as the recall mechanism.

### Hybrid recall→precision (RECOMMENDED PIPELINE SHAPE)
- Stage 1 (recall, cheap): embeddings + cosine top-K nearest neighbors → candidate pairs.
- Stage 2 (precision, expensive, OPTIONAL): Claude judges only those candidates: "do these compute the same thing? yes/no + 1-line why."
- This is the consensus modern design (HyClone, VulCoCo): cheap embedding recall, expensive verification only on survivors. [CITED: arxiv.org/pdf/2508.01357 HyClone]

## 3. Recommended MVP

**Shape:** Local embeddings recall + cosine threshold + ranked report. Claude-judge is a **flagged-off Stage 2** you can enable later. No server, no API key required for the MVP.

### Extraction (what to embed)
- **Function/method-level granularity.** Each top-level function, method, and arrow-function-assigned-to-const becomes one unit. Block-level adds noise and explodes count; function-level matches how GSD duplication actually happens (a whole helper re-written).
- **Parser: `web-tree-sitter`** (WASM, pure Node, no native build) with `tree-sitter-javascript` and `tree-sitter-typescript` grammars. One query pulls `function_declaration`, `method_definition`, arrow functions bound to declarators. [VERIFIED: npm `web-tree-sitter@0.26.9`; tree-sitter multi-language extraction is standard practice]
- **Markdown (`workflows/`, `skills/`): OUT of MVP scope.** Those are prose/process specs, not executable logic; embedding them invites cross-domain false positives. Note as a deferred extension (could embed per-section to catch duplicated *instructions*, but that's a different problem).
- Store per unit: `{ id, file, startLine, endLine, lang, name, sourceText, tokenHash }`.

### Chunking / multi-language
- One unit = one function, truncated to model's 8192-token window (functions over that are rare; flag+skip). Embed each independently. Mixing CJS and TS in one vector space is fine — `jina-...-code` is multilingual and the point is cross-file/cross-runtime recall.

### Similarity + threshold guidance
- Cosine over L2-normalized mean-pooled embeddings.
- **Thresholds are model-specific and must be tuned, not assumed.** [CITED: threshold survey — CodeBERT optimal ~0.4, BGE-M3 ~0.6, others vary]. For `jina-...-code`, **start at cosine ≥ 0.80, report the 0.80–0.97 band, and exclude ≥ 0.985 as likely textual/Type-1 (already caught by jscpd).** Treat the first run as calibration: eyeball the ranked list and move the floor. [ASSUMED — starting band; needs empirical calibration on this repo]
- **Cosine-saturation caveat:** code embeddings cluster high (many pairs ~0.85); rank by similarity and review the *top of the list*, don't trust the absolute number as a hard pass/fail. [CITED: dev.to "Cosine Similarity Lies"]
- Exclude pairs whose normalized-token hashes are equal (those are Type-1/2 — that's jscpd's job, not ours).

### Intentional-duplication allowlist (REQUIRED — first-class concept)
The repo's **dual resolver is a deliberate CJS↔SDK duplicate** (`bin/lib/*.cjs` ↔ `sdk/src/**`) and MUST NOT be reported as a defect. Design:
- A committed `**.semdup-allow.json`** (or `.semdupignore`) listing intentional-dup rules. Each rule matches a **pair** by glob, not a single file, e.g.:
  ```json
  { "intentional": [
    { "a": "bin/lib/**", "b": "sdk/src/**",
      "reason": "Dual CJS/SDK runtime resolver — duplicated on purpose" }
  ] }
  ```
- Before emitting a suspect pair, suppress it if `(a matches rule.a && b matches rule.b)` or the symmetric case. Suppressed pairs are written to a separate `--show-suppressed` view (never silently dropped — auditability).
- Also support a per-function opt-out marker comment (`// semdup-allow: <reason>`) for one-off intentional clones.
- **Principle:** an intentional-dup escape hatch is not optional polish — without it the tool's first run drowns in the known CJS/SDK pairs and gets ignored.

### Output format
Ranked Markdown + JSON report of suspect pairs:
```
### Suspect #1  — cosine 0.91
A: bin/lib/config.cjs:142–171  resolveModel()
B: sdk/src/query/state-mutation.ts:88–120  pickModel()
Why: both map a config key → model id with the same fallback chain
[ not in allowlist ]  [ token-overlap: 12% → genuinely Type-4 ]
```
JSON carries `{aFile,aLines,bFile,bLines,cosine,tokenOverlap,suppressed,reason}` for later `/gsd:` consumption.

## 4. Pitfalls

- **False positives — boilerplate:** getters/setters, trivial wrappers, `module.exports` shims, 3-line guard clauses all embed near each other. Mitigate: **min function size** (skip < ~5 statements / < ~120 tokens), and report token-overlap % alongside cosine so reviewers can spot "structurally trivial" pairs.
- **False positives — intentional dual-impl:** handled by the allowlist above (the #1 repo-specific risk).
- **False positives — generated/test code:** exclude `*.generated.*`, `tests/**`, `**/fixtures/**`, `node_modules`, `dist/`, `sdk/dist/` by default. (Repo already has `command-aliases.generated.cjs`.)
- **Threshold tuning:** absolute cosine is not portable; calibrate per-repo, rank don't gate. ROC-style eyeballing on first run.
- **Cross-language equivalence (CJS↔TS *semantics*):** the model *can* match across JS and TS, which is what we want for the dual-resolver class — but genuine cross-*paradigm* equivalence (e.g. a bash workflow vs a TS function) is **explicitly OUT of MVP scope.** Say so.
- **Performance:** ~3k functions. Embedding 3k short snippets on CPU at q8 ≈ minutes (one-time, cache embeddings to disk keyed by tokenHash). Pairwise cosine is 3k² ≈ 9M float-dot ops — trivial in-memory, but use **`hnswlib-node`** (ANN index) if it grows past ~20k units to keep top-K query fast. [VERIFIED: npm `hnswlib-node` passes slopcheck OK]. Cache invalidation by content hash means re-runs only re-embed changed functions.

## 5. Build plan

**Runtime:** Node (fits the repo; later promotable to a `/gsd:` command). Standalone runnable script for MVP.

**Key libraries (all slopcheck `[OK]`, versions verified):**
| Package | Version | Role |
|---------|---------|------|
| `@huggingface/transformers` | 4.2.0 | local ONNX embeddings (jina-...-code, q8) [VERIFIED: npm] |
| `web-tree-sitter` + grammars | 0.26.9 | WASM AST function extraction, no native build [VERIFIED: npm] |
| `jscpd` | 5.0.11 | optional cheap Type-1/2 pre-pass to exclude textual clones [VERIFIED: npm] |
| `hnswlib-node` | (latest) | optional ANN index if unit count grows [VERIFIED: slopcheck OK] |

> Provenance note: model name `jina-embeddings-v2-base-code` is [CITED: huggingface.co model card]; all package names verified on npm AND slopcheck-clean, but treat the embedding *quality* claims as MEDIUM until calibrated on this repo.

**File layout (standalone tool):**
```
tools/semdup/
├── extract.mjs       # tree-sitter → function units (JS/CJS/TS)
├── embed.mjs         # @huggingface/transformers, mean-pool + L2 norm, disk cache
├── compare.mjs       # cosine top-K, token-overlap, threshold band
├── allowlist.mjs     # load .semdup-allow.json, suppress intentional pairs
├── report.mjs        # ranked Markdown + JSON
├── index.mjs         # pipeline orchestrator + CLI flags
└── .semdup-allow.json
```

**Pipeline:** `extract → (jscpd exclude exact) → embed(cache) → cosine top-K → drop allowlisted → rank → report`.

**Command:**
```bash
node tools/semdup/index.mjs --root . \
  --include 'bin/**/*.cjs,sdk/src/**/*.ts' \
  --min-tokens 120 --cosine-floor 0.80 \
  --allowlist tools/semdup/.semdup-allow.json \
  --out .planning/semdup-report.md
# later, opt-in precision pass:
#   --judge claude   (Stage 2: LLM verifies top candidates)
```

**Later → `/gsd:` command:** wrap `index.mjs`, default the allowlist to the repo's, write report into `.planning/`, and have the Stage-2 Claude-judge produce the "why equivalent" lines for the top N pairs only.
